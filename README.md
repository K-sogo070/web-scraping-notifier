# ウェブスクレイピング＆通知システム (Web Scraping Notifier)

## 概要

指定したWebサイト群を定期的に自動巡回し、過去データとのハッシュ値比較によって新着情報のみを検知、Slackへ非同期で通知する監視集約システムです。

単なるCRUD（データの出し入れ）アプリではなく、以下のような「実務に直結する技術要素」を意識して設計・実装しました。

- 定期実行によるバッチ的な自動巡回処理
- ハッシュ値比較による効率的な差分検知アルゴリズム
- 非同期処理による通知処理の分離
- robots.txt遵守やエラーハンドリングなど、法務・運用面への配慮

## 主な機能

| 機能 | 説明 |
|---|---|
| 定期巡回 | `@Scheduled`により、登録された監視対象サイトを一定間隔で自動巡回 |
| 差分検知 | 記事単位でSHA-256ハッシュ値を計算し、DB未登録のものだけを新着として抽出 |
| Slack通知 | 新着記事を検知すると、`@Async`による非同期処理でSlackへ即座に通知 |
| 監視対象の動的管理 | 監視対象サイトをコードにハードコードせず、DBのマスタテーブルで管理 |
| robots.txt遵守 | クロール前に対象サイトのrobots.txtを解析し、許可されたサイトのみ巡回 |

## 技術スタック

- **バックエンド**: Java 21, Spring Boot 3.5
- **データアクセス**: Spring Data JPA (Hibernate)
- **データベース**: PostgreSQL 18
- **スクレイピング**: Jsoup
- **通知連携**: Slack Incoming Webhook
- **非同期・定期実行**: Spring `@Async` / `@Scheduled`
- **バージョン管理**: Git / GitHub

## アーキテクチャ

```
[監視対象のWebサイト群]
        │ (定期的に巡回)
        ▼
[① robots.txt遵守チェック]
        │
        ▼
[② クローラー：Jsoupで対象サイトを巡回・HTML解析]
        │
        ▼
[③ 差分検知：過去データとハッシュ値比較]
        │ (新着ありの場合)
        ▼
[④ DB保存：monitor_targets / scraped_items / notification_logs]
        │
        ▼
[⑤ 非同期通知：@AsyncでSlackへ送信]
```

### DB構成

- **monitor_targets**：監視対象サイトのマスタ（URL・CSSセレクタ・巡回間隔・robots.txt遵守フラグ等）
- **scraped_items**：収集した記事データ（タイトル・URL・ハッシュ値等）
- **notification_logs**：通知の送信履歴（成功／失敗の記録）

## 工夫した点

### 1. 拡張性を意識したDB設計
監視対象サイトをコードにハードコードせず、`monitor_targets` テーブルで管理する設計にしました。これにより、**コードの変更・再デプロイを一切行わずに、DBへのINSERT操作だけで新しい監視対象サイトを追加できます**。実際にサカナクション公式サイトを監視対象として追加した際も、SQL文を1件実行するだけで対応できました。

「機能追加のたびにコードを書き換える」設計と比べ、運用時の柔軟性・保守性を大きく高められる点を意識しました。

### 2. 法務・運用リスクへの配慮
Webスクレイピングは技術的に実装できても、相手サーバーへの配慮を怠ると実務では通用しません。以下を実装しています。
- クロール前に対象サイトの `robots.txt` を解析し、禁止されている場合はスキップ
- サーバー負荷軽減のため、1サイト巡回ごとに `Thread.sleep(1000)` で間隔を空ける
- User-Agentに連絡先情報を含め、クローラーとしての身元を明示

### 3. 堅牢なエラーハンドリング
1つの監視対象サイトでエラー（サイト構造の変更、タイムアウト等）が発生しても、他のサイトの処理には影響しないよう、`try-catch`で個別に例外を処理し、ログを残して処理を継続する設計にしています。

### 4. 差分検知アルゴリズムの選定
記事単位でSHA-256ハッシュ値を計算し保持する方式（記事単位ハッシュ方式）を採用しました。ページ全体のハッシュを比較する方式と比べ、「どの記事が新着か」を個別に特定できる点を優先しています。

## 苦労した点・学んだこと

### 文字化けの原因調査(表示上の問題とデータ上の問題の切り分け)
開発中、コンソールのログ上で一部のタイトルが `?` に文字化けする現象が発生しました。最初は「スクレイピングしたデータ自体が壊れているのでは」と考えましたが、以下の手順で原因を切り分けました。

1. `Document.charset()` でJsoupが検出した文字コードを確認 → `UTF-8` と正しく検出されていることが判明
2. pgAdmin 4で直接DBの中身を確認 → 保存されているデータ自体は正しい文字列であることを確認

この2段階の確認により、**原因はWindowsのコンソールの表示上の問題であり、実際のデータやロジックには問題がなかった**と結論づけました。「バグに見える現象」が実際に問題のある箇所で起きているとは限らず、表示・保存・処理のどの段階で問題が起きているかを切り分けて考える重要性を学びました。

## 環境構築手順

### 前提
- Java 21以上
- PostgreSQL 18
- Maven（プロジェクトに同梱の `mvnw` を使用可能）

### 手順

1. リポジトリをクローン
git clone https://github.com/K-sogo070/web-scraping-notifier.git

2. PostgreSQLで `portfolio` という名前のデータベースを作成

3. `src/main/resources/application.properties.example` をコピーし、`application.properties` にリネーム
   - `spring.datasource.password` に自分のPostgreSQLパスワードを設定
   - `notification.slack.webhook-url` に自分のSlack Incoming Webhook URLを設定

4. アプリを起動
./mvnw spring-boot:run
